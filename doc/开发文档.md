# OKX多Agent自动交易机器人 开发文档（面向LLM开发，MVP版本）
## 1. 文档概述
### 1.1 文档目的
本文档为LLM开发提供完整的需求、架构、规范指导，确保开发出的产品完全符合用户需求，无代码实现细节，仅定义边界、规则、流程与约束。
### 1.2 MVP边界说明
本版本为最小可行性产品，仅包含核心交易能力，以下内容暂不实现，后续迭代补充：
- 配置热加载
- 分析历史记录持久化（仅内存存储）
- OKX私有频道WebSocket订阅（仅用REST接口查询数据）
- 交易策略规则校验（完全依赖Agent Prompt约束）
- 多账户支持
- 消息通知功能
- 性能监控面板
- 策略回测功能
### 1.3 参考文档
- OKX官方API V5文档：https://www.okx.com/docs-v5/zh/
- OpenAI官方Python SDK文档
## 2. 需求拆解
### 2.1 功能需求
1. **自主OKX客户端**：完全自行封装OKX REST/WebSocket连接，禁止使用任何第三方OKX SDK/库
2. **多Agent架构**：包含分析师、交易员、压缩者三个独立Agent，基于OpenAI SDK实现
3. **配置体系**：
   - `.env`存储敏感密钥（OKX实盘/模拟盘密钥、LLM密钥）
   - `config.toml`存储程序配置，支持多交易对独立配置、并行运行
   - `prompt.toml`统一存储所有Agent的System Prompt
   - 支持`demo_mode`切换，开启时自动使用模拟盘密钥连接OKX模拟环境
4. **启动自检**：启动时自动校验交易所连接、LLM连接、LLM图片解析能力，校验失败直接退出
5. **日志系统**：日志同时输出到终端和文件，文件名精确到秒，支持级别配置
6. **事件驱动主逻辑**：
   - 基于OKX WebSocket监听K线收盘事件，禁止轮询
   - K线收盘后获取N根历史K线，计算EMA20，生成带坐标轴、EMA20均线的K线图，K线图配色为黑白模式，阳线是黑色边框、黑色影线、白色实体（实体在影线图层之上），阴线是纯黑色。k线图上有周期和交易对信息。
   - 将K线图、分析历史、账户余额、当前仓位发送给分析师
   - 并行调用交易员（输出JSON交易指令数组）和压缩者（压缩分析结果存储历史）
   - 支持批量执行多条交易指令
   - 平仓后自动清空分析历史、写入交易记录到CSV
   - 分析历史超出最大长度时自动丢弃最早记录
7. **交易规则**：
   - 启动时自动设置所有交易对为单向持仓模式
   - 支持7种交易指令：`entry_long`/`entry_short`（市价开仓附带止盈止损）、`close_position`（全平并自动取消止盈止损）、`change_stop`/`change_profit`（修改止盈止损）、`exit_long`/`exit_short`（仅减仓不平仓）
### 2.2 非功能需求
1. **硬件适配**：可在2核CPU、2G内存的设备上稳定运行
2. **性能要求**：
   - 无关联任务全部并行处理，减少等待耗时
   - 网络请求高效，复用连接，避免无效请求
   - 单交易对分析流程耗时不超过1分钟（含LLM调用）
   - 支持至少5个交易对并行运行
3. **稳定性要求**：异常场景下不崩溃、不重复下单、不出现资损
## 3. 系统架构设计
采用分层异步架构，基于Python asyncio实现，充分利用IO多路复用降低资源占用，适配低配置硬件。
### 3.1 分层架构
| 层级       | 职责描述                                                                                                  |
| ---------- | --------------------------------------------------------------------------------------------------------- |
| 应用层     | 主程序入口、启动自检、事件循环管理、优雅关闭逻辑                                                          |
| 服务层     | 交易服务、K线服务、Agent调度服务、历史记录管理服务、交易记录存储服务                                      |
| 领域层     | 交易领域模型（K线、仓位、订单、指令）、Agent模型、事件模型                                                |
| 基础设施层 | 自主实现的OKX REST/WebSocket客户端、LLM客户端、日志组件、配置加载组件、工具组件（绘图、EMA计算、CSV操作） |
### 3.2 运行模型
1. 整体基于异步协程实现，避免多进程/多线程的内存/CPU开销，适配2核2G硬件
2. 每个交易对对应独立的协程，互不影响，实现并行运行
3. 所有IO操作（网络请求、文件读写）全部采用异步实现
### 3.3 并行设计
1. 多个交易对的分析流程完全并行
2. 单交易对分析流程中，查询余额、查询仓位两个无关联请求并行执行
3. 分析师返回结果后，交易员、压缩者的调用并行执行
## 4. 基础设施层规范
### 4.1 OKX客户端封装规范
**禁止使用任何第三方OKX相关库，完全基于aiohttp自主实现**
1. **端点切换**：
   - 实盘：REST端点`https://www.okx.com/api/v5`，WebSocket公共端点`wss://ws.okx.com:8443/ws/v5/public`
   - 模拟盘：REST端点同实盘，请求头添加`x-simulated-trading: 1`，WebSocket公共端点`wss://wspap.okx.com:8443/ws/v5/public`
2. **签名实现**：严格按照OKX官方文档实现HMAC-SHA256签名算法，签名内容包含`timestamp + method + requestPath + body`
3. **REST客户端要求**：
   - 复用HTTP连接池，减少TCP握手开销
   - 超时时间设置为10秒，支持指数退避重试
   - 自动处理签名、请求头拼装
   - 实现以下接口（对应用户提供的官方API文档）：
     - 账户类：查询余额、查询仓位、设置持仓模式、设置杠杆、查询交易对信息
     - 交易类：下单（附带止盈止损）、修改订单、平仓、查询订单详情
     - 行情类：查询历史K线
4. **WebSocket客户端要求**：
   - 支持自动重连，重连时自动重新订阅所有K线频道
   - 最多重连5次，间隔指数退避（1s/2s/4s/8s/16s），重连失败触发程序退出
   - 订阅K线频道，解析推送数据的`confirm`字段，`confirm=1`时触发K线收盘事件
### 4.2 LLM客户端封装规范
1. 基于OpenAI官方异步Python SDK实现
2. 超时时间设置为30秒，支持1次重试
3. 统一处理API Key、Base URL配置，支持多模态调用（图片解析）
4. 调用时`temperature`参数默认设为0，保证输出确定性（压缩者可设为0.1）
### 4.3 日志组件规范
1. 同时输出到终端和文件，日志级别支持`DEBUG/INFO/WARNING/ERROR/FATAL`
2. 日志文件名格式：`{log_dir}/trade_bot_{YYYYMMDD}_{HHMMSS}.log`，精确到秒
3. 日志格式：`[YYYY-MM-DD HH:MM:SS] [级别] [模块名] 内容`
4. 交易操作必须记录审计日志，包含操作参数、执行结果、订单ID
5. 错误日志必须包含完整异常栈信息
### 4.4 配置加载规范
1. 启动时一次性加载所有配置，MVP不支持热加载
2. `.env`文件使用`python-dotenv`加载，禁止提交到版本库
3. TOML文件使用标准库`tomllib`（Python 3.11+）或`tomli`解析
### 4.5 工具组件规范
1. **EMA计算**：基于收盘价计算标准EMA20，平滑系数为`2/(20+1)`，使用numpy向量化运算提高效率
2. **K线绘图**：
   - 生成PNG格式图片，分辨率1080p
   - 包含清晰坐标轴：X轴为时间（格式`YYYY-MM-DD HH:MM`），Y轴为价格
   - 阳线是黑色边框、黑色影线、白色实体（实体在影线图层之上），阴线是纯黑色，EMA20均线为蓝色
   - 包含k线周期和交易对信息
   - 无多余装饰元素，生成后仅保存在内存中，转base64后立即释放绘图对象，避免内存泄漏。测试中允许保存到本地目录
3. **CSV存储**：平仓记录追加写入CSV，支持自动创建文件和表头
## 5. 领域层规范
### 5.1 交易领域模型
1. **K线模型**：字段包含`timestamp`（时间戳）、`open`/`high`/`low`/`close`（OHLC价格）、`vol`（成交量）、`confirm`（是否收盘）
2. **仓位模型**：字段包含`inst_id`（交易对ID）、`direction`（`long/short/empty`）、`size`（持仓数量）、`entry_price`（开仓均价）、`stop_price`（当前止损价）、`profit_price`（当前止盈价）、`unrealized_pnl`（未实现盈亏）
3. **交易指令模型**：字段包含`op`（操作类型，仅支持规定的7种）、`args`（参数字典）、`client_oid`（UUID生成的幂等ID，防止重复下单）
4. **交易记录模型**：字段包含`timestamp`、`inst_id`、`position_direction`、`position_size`、`entry_avg_price`、`exit_avg_price`、`realized_pnl`、`balance_after_close`、`order_id`
### 5.2 Agent模型
所有Agent无状态，每次调用传入完整上下文：
1. **分析师**：
   - 输入：K线图base64、分析历史记录、当前仓位、账户余额、交易对配置、当前时间
   - 输出：自然语言分析结果与交易决策
2. **交易员**：
   - 输入：分析师输出（含方向、止损、止盈）、当前仓位、账户余额、当前价格、单笔风险百分比、交易对配置
   - 输出：严格符合JSON Schema的交易指令数组，无任何多余文字
   - 开仓数量计算逻辑：根据分析师给出的方向和止损价格，结合当前价格、账户余额、单笔风险百分比，使用公式 `开仓数量 = (账户余额 × 单笔风险百分比) ÷ |当前价格 - 止损价格| × 当前价格` 计算开仓数量
3. **压缩者**：
   - 输入：分析师输出
   - 输出：不超过100字的压缩文本，保留核心决策逻辑（方向、止损、止盈、核心依据）
### 5.3 事件模型
1. **K线收盘事件**：包含`inst_id`、`timeframe`、收盘K线数据
2. **交易完成事件**：包含`inst_id`、`op`、`order_id`、执行结果、错误信息
3. **平仓完成事件**：包含`inst_id`、平仓仓位信息、平仓后账户余额
## 6. 服务层规范
### 6.1 交易服务规范
1. 启动时必须调用OKX接口，查询持仓模式，如果持仓模式不为单向持仓，则将所有交易对设置为**单向持仓模式**（`posMode=net`），并设置对应杠杆。
2. 所有交易操作前必须重新查询最新仓位和余额，不依赖本地缓存，避免数据不一致
3. 指令参数校验规则：
   - `entry_long`/`entry_short`：止损止盈价格必须符合配置的比例要求，开仓金额不超过配置的仓位大小
   - `exit_long`/`exit_short`：减仓数量必须小于当前持仓数量，禁止全平（全平必须用`close_position`）
   - `change_stop`/`change_profit`：必须持有对应方向的仓位
4. 开仓时必须通过OKX的`attachAlgoOrds`参数附带止盈止损条件单，避免单独挂单的时序风险
5. 平仓必须调用OKX官方平仓API，自动取消对应止盈止损订单
6. 所有交易请求必须携带`client_oid`实现幂等，避免重复下单
### 6.2 K线服务规范
1. K线收盘事件仅通过WebSocket推送触发，禁止轮询
2. 收到K线收盘事件后，调用REST接口获取最近`k_line_count`根已收盘的K线数据，确保数据完整
3. 计算EMA20后生成K线图，转base64编码用于LLM调用
### 6.3 Agent调度服务规范
1. 所有Agent调用为异步实现，交易员与压缩者的调用并行执行
2. 调用前自动替换Prompt中的占位符（如`{inst_id}`、`{position}`等）
3. 必须在system prompt中显式约束交易员的输出格式，并在程序内对交易员的输出进行JSON Schema校验，校验失败触发重试
### 6.4 历史记录管理服务规范
1. 每个交易对维护独立的分析历史记录列表，仅存储在内存中
2. 新增压缩后的分析记录时，若长度超过`max_analysis_history_length`，自动删除最早的1条
3. 收到平仓完成事件后，立即清空对应交易对的所有分析历史记录
## 7. 应用层规范
### 7.1 启动自检流程
所有步骤校验失败直接退出，输出错误日志，返回对应退出码：
1. 加载`.env`、`config.toml`、`prompt.toml`，校验配置格式合法性（退出码：2）
2. 初始化OKX客户端，调用查询余额接口验证连接有效性，校验所有配置的交易对ID合法性（退出码：3）
3. 调用OKX接口设置单向持仓模式、设置每个交易对的杠杆，验证操作成功（退出码：3）
4. 初始化LLM客户端，发送测试文本请求（要求返回`OK`），验证LLM连接有效性（退出码：3）
5. 生成测试K线图，发送给LLM要求描述趋势，验证LLM图片解析能力（退出码：3）
6. 自检通过，输出启动日志，进入主事件循环
### 7.2 主事件循环流程
1. 为每个交易对启动独立协程，订阅对应时间周期的K线WebSocket频道
2. 监听K线收盘事件，触发对应交易对的分析流程
3. 监听系统终止信号（SIGINT/SIGTERM），等待正在运行的分析流程完成，关闭所有连接，刷新日志后优雅退出
### 7.3 单交易对分析流程
1. 尝试获取交易对专属的异步互斥锁，获取失败（上一次分析未完成）则丢弃本次触发，记录警告日志，流程终止
2. 并行调用OKX接口查询当前账户余额、当前交易对仓位
3. 获取最近N根K线数据，计算EMA20，生成K线图base64
4. 组装上下文，调用分析师Agent获取分析结果
5. 并行执行：
   - 调用交易员Agent获取交易指令数组，校验格式合法性，重试最多3次
   - 调用压缩者Agent获取压缩后的分析结果，存入分析历史
6. 逐条执行交易指令，记录执行结果
7. 若执行了`close_position`指令，触发平仓完成事件，写入交易记录到CSV，清空分析历史

## 9. 异常处理与稳定性规范
### 9.1 已知问题处理方案
1. **前一次分析未完成时新K线收盘**：每个交易对配备异步互斥锁，获取锁失败则丢弃本次触发，记录警告日志，避免任务堆积
2. **交易员输出错误JSON格式**：
   - 用JSON Schema校验输出，校验失败时将错误信息返回给交易员重试
   - 最多重试3次，间隔1秒
   - 3次失败则丢弃本次指令，记录错误日志，不执行任何操作
3. **网络波动请求失败**：采用指数退避重试，最多3次，间隔1s/2s/4s，失败则终止本次流程，记录错误日志
4. **WebSocket断开**：自动重连并重新订阅频道，最多5次重试，失败则程序退出
5. **交易执行失败**：记录错误日志，继续执行后续指令（若有），终止本次流程
### 9.2 错误分级
| 错误级别 | 处理逻辑                   | 退出码 |
| -------- | -------------------------- | ------ |
| 致命错误 | 程序立即退出，打印错误日志 | 非0    |
| 严重错误 | 终止本次流程，记录错误日志 | 无     |
| 警告     | 记录警告日志，继续运行     | 无     |
### 9.3 重试机制
- OKX REST请求：最多3次指数退避重试
- LLM请求：最多1次重试，超时30秒
- WebSocket重连：最多5次指数退避重试
- 交易员JSON格式错误：最多3次重试
## 10. 性能与资源约束规范
### 10.1 硬件适配要求
- 最低配置：2核CPU、2G内存、1M带宽
- 运行时资源限制：内存占用峰值≤1G，CPU占用峰值≤70%
### 10.2 内存优化要求
- K线数据仅保留需要的N根，用完立即释放
- K线图仅存内存，转base64后立即释放绘图对象
- 分析历史仅保留最大长度条目，超出自动删除
- 避免全局变量，所有对象使用局部变量，便于GC回收
### 10.3 网络优化要求
- 复用HTTP/WebSocket连接，减少握手开销
- 无关联请求并行执行，减少总耗时
- 禁止无效轮询，所有事件基于推送触发
